# StudyHelper 测试流程规范

## 一、测试闭环流程

```
开发完成
    ↓
1. 单元测试（开发自己跑）
    ↓ 产出：单元测试报告
2. 功能测试（QA执行）
    ↓ 产出：测试用例执行结果
    ├─ 通过 → 进入下一轮
    └─ 失败 → 提交缺陷报告 → 打回开发修复
    ↓
3. 回归测试（修复后重新验证）
    ↓ 产出：回归测试报告
4. 测试闭环确认
    ↓
5. 发布 + 代码上传 GitHub
```

## 三、缺陷管理

### 缺陷状态流转
```
新建 → 分配给开发 → 开发修复中 → 已修复 → 测试验证
                                                    ↓
                                              验证通过 → 关闭
                                              验证失败 → 打回重新修复
```

### 缺陷记录格式
```markdown
| 缺陷ID | 模块 | 描述 | 严重程度 | 状态 | 负责人 | 创建时间 | 关闭时间 |
|--------|------|------|---------|------|--------|---------|---------|
| BUG-001 | 知识库 | 知识点收藏失败 | P2 | 已关闭 | 王浩然 | 05-11 | 05-11 |
```

### 缺陷闭环标准
- 所有 P0/P1 缺陷必须关闭
- P2 缺陷关闭率 ≥ 90%
- 无阻塞性问题

## 四、测试环境

| 环境 | 用途 | 数据 |
|------|------|------|
| Dev | 开发自测 | 模拟数据 |
| Test | 功能测试 | 测试数据 |
| Staging | 回归测试 | 脱敏生产数据 |
| Production | 最终发布 | - |

## 五、发布标准

### 测试闭环检查清单
- [ ] 前端单元测试覆盖率 ≥ 60%
- [ ] 后端单元测试覆盖率 ≥ 70%
- [ ] 功能测试用例执行率 100%
- [ ] 冒烟测试通过率 100%
- [ ] 所有 P0/P1 缺陷已关闭
- [ ] P2 缺陷关闭率 ≥ 90%
- [ ] 回归测试无新问题引入
- [ ] 性能测试通过（可选）

### 发布流程
```
测试闭环确认
    ↓
代码冻结（禁止新功能代码合并）
    ↓
生成测试报告
    ↓
提交代码到 GitHub
    ↓
部署到生产环境
    ↓
发布完成
```
